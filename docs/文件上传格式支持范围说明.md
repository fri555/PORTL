# MVP 阶段上传格式支持范围与依据

> 本文为 `docs/test/知识库测试用例.md` 的补充说明，用于指导 MVP-1（v1.7.0）实施时确定最小可用格式集合，并明确后续阶段（增强-3 / v1.11.x 多模态）的扩展路线。

---

## 1. 推荐范围

MVP-1 阶段建议支持 **9 种核心格式**，按"必选 / 强烈建议 / 选做"三层划分：

| 优先级 | 格式 | 后缀 | 用例覆盖 | 工程成本 | 业务价值 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **P0 必选** | PDF（文本型） | `.pdf` | TC-DS-001、TC-MM-001~003 | 中 | ⭐⭐⭐⭐⭐ 企业知识主载体 |
| **P0 必选** | Word | `.docx` | TC-DS-002、TC-MM-004 | 低 | ⭐⭐⭐⭐⭐ 制度/规范主载体 |
| **P0 必选** | TXT | `.txt` | TC-DS-003、TC-KP-001/002 | 极低 | ⭐⭐⭐⭐ 导出的纯文本/日志 |
| **P0 必选** | Markdown | `.md` | TC-DS-004、TC-KP-003 | 极低 | ⭐⭐⭐⭐ 技术/产品文档主载体 |
| **P1 强烈建议** | Excel | `.xlsx` | TC-DS-006、TC-MM-005 | 中 | ⭐⭐⭐⭐ 财务/数据主载体 |
| **P1 强烈建议** | 扫描 PDF（OCR） | `.pdf`（扫描件） | TC-DS-024~025、TC-MM-010/011 | 高 | ⭐⭐⭐⭐ 历史纸质档案 |
| **P1 强烈建议** | PNG / JPG | `.png`/`.jpg` | TC-DS-005、TC-KP-009、TC-MM-010~012 | 中 | ⭐⭐⭐ 截图/票据/图表 |
| **P2 选做** | PPT | `.pptx` | TC-MM-006 | 中 | ⭐⭐⭐ 培训/汇报资料 |
| **P2 选做** | CSV | `.csv` | （需新增用例） | 极低 | ⭐⭐ 数据导出/对接 |

> **MVP 暂不支持**（推迟到增强-3 / v1.11.x 多模态阶段）：`.doc`（旧版 Word）、`.xls`（旧版 Excel）、`.rtf`、音视频（MP3/MP4）、`.zip` 压缩包、`.html`、邮件 `.eml`、流程图专用格式（`.drawio`/`.vsdx`）。

## 2. 选型依据

### 2.1 业务侧：覆盖企业 80%+ 知识资产

| 维度 | 结论 |
| :--- | :--- |
| **企业知识主载体** | PDF + Word 覆盖大多数制度、规章、产品手册；Markdown 覆盖技术/产品文档；Excel 覆盖财务/运营数据 |
| **历史数据迁移** | 扫描 PDF OCR 解决存量纸质档案数字化，是"上线即用"的关键 |
| **多模态基线** | PNG/JPG 覆盖截图、票据、图表，是后续 OCR/图表理解/多模态检索的基础 |

> **经验法则**：企业知识库 80% 内容集中在 PDF / Word / Excel / 扫描件四类，9 种格式组合可覆盖约 90% 业务场景。

### 2.2 工程侧：成本与价值平衡

| 格式 | 解析器生态 | 处理链路 | 成本评估 |
| :--- | :--- | :--- | :--- |
| TXT / MD | Python 内置 + `markdown` 库 | 分块→向量化→入库 | 极低，1 人天 |
| PDF（文本） | `pypdf` / `pdfplumber` / `unstructured` | 同上 | 中，3~5 人天 |
| Word | `python-docx` | 同上 | 低，1~2 人天 |
| Excel | `openpyxl` / `pandas` | 表结构化→分块→入库 | 中，3~5 人天 |
| 扫描 PDF OCR | `paddleocr` / `tesseract` + 文本型PDF流水线 | OCR→分块→入库 | 高，5~10 人天 |
| PNG/JPG | `pillow` + OCR + CLIP | OCR/视觉编码→入库 | 中，3~5 人天 |
| PPT | `python-pptx` | 同 Word | 中，3~5 人天 |

> **总投入估算**：MVP 9 种格式约 20~35 人天，相对一个迭代（约 10 个工作日 × 团队规模）可控。

### 2.3 风险侧：规避高投入低产出

| 推迟项 | 推迟原因 |
| :--- | :--- |
| `.doc` 旧版 Word | 占比 <5%，企业已基本完成升级；解析需 `antiword`/`libreoffice`，生态弱 |
| `.xls` 旧版 Excel | 同上，可用 `.xlsx` 转换；`xlrd` ≥2.0 已不读 `.xls` |
| MP3/MP4 音视频 | 需 ASR 引擎（Whisper/Paraformer），算力成本高，且与 RAG 检索协同弱，建议增强-3 阶段 |
| `.zip` 压缩包 | 安全风险大（ZIP Bomb），需多层防护，优先 LPA 检查再考虑 |
| 邮件 `.eml` | 解析需处理 MIME 嵌套、附件关系，链路复杂，按需排期 |

## 3. 实施建议

### 3.1 MVP-1（v1.7.0）必做
1. **P0 四种格式** 端到端跑通：PDF（文本）/ Word / TXT / Markdown，覆盖 TC-DS-001~004
2. **格式校验**：用文件头魔数（`python-magic`）而非扩展名，规避伪装
3. **统一流水线**：解析→分块（512 tokens + 50 overlap）→向量化→三索引入库
4. **监控埋点**：按格式分桶统计成功率、耗时、chunk 数

### 3.2 MVP-1（v1.7.1）补齐
1. **P1 三种格式**：Excel + 扫描 PDF OCR + PNG/JPG，覆盖 TC-DS-005/006/024/025
2. **OCR 降级策略**：扫描件置信度 <0.8 标记 `quality=low`，进入人工审核队列
3. **多模态向量**：图片走 CLIP/BLIP，统一向量空间

### 3.3 增强-3（v1.11.0~11.1）扩展
1. **P2 格式**：PPT + CSV
2. **多模态深化**：音视频 ASR、图表理解、公式识别（覆盖 TC-MM-007~012）
3. **历史归档格式**：`.doc`/`.xls` 兼容模式（按需评估投入产出比）

## 5. 总结

> **MVP 阶段格式策略**：
> - **3 天铺 4 种**（TXT/MD/PDF/Word）→ 系统可用
> - **+5 天补 3 种**（Excel/扫描PDF/图片）→ 业务可用
> - **后续按需扩展**（PPT/CSV/音视频）→ 业务增值
>
> 这一节奏与 1 个月 2 个迭代的 MVP 计划匹配：**v1.7.0 跑通主链路，v1.7.1 覆盖企业 90% 场景，v1.11.x 达到行业领先水平**。
