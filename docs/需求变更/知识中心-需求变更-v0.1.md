# 需求变更

> 基于PRD（20260703-【PRD】AI工作台-知识中心-天马智擎项目.md）与当前原型实现（frontend-mvp/src/views/KnowledgeBaseView.vue）之间的差距分析，逐项记录已修复和待修复的变更点。

---

## 文件树左侧边栏

### 需求1: 展开折叠分离

**PRD对照（§3.2.1.1 EARS事件驱动型）：**「当用户点击折叠按钮时，系统应以 translate-x 动画收起左侧栏」及文件树展开/折叠交互属于基本导航行为，小三角应仅切换展开状态，不触发主视图导航。

**当前差距：** `toggleTreeNode()`同时做了设置 `activeTreeId/selectedKbId`（导航）和切换 `expandedTreeIds`（展开/折叠），导致点击小三角后主视图跳转，展开效果不可见。

**变更内容：**
- 新增 `toggleTreeExpandOnly(node)`，仅切换 `expandedTreeIds`
- 文件树中两处知识库/文件夹小三角的 `@click` 改为 `toggleTreeExpandOnly`
- `toggleKbTreeExpand()` 内部改为调用 `toggleTreeExpandOnly`
- `toggleTreeNode()` 保留给主内容区文件夹行使用（列表视图/卡片视图的文件夹点击）

### 需求2: Mock数据层级重构

**PRD对照（§3.1.2 层级结构 + §3.2.1 EARS普遍型）：** PRD规定：
- 空间（公共空间/个人空间）→ 空间文件夹 → 知识库 → 文件夹 → 文件的五级结构
- 同一文件只允许挂在一个位置，不能在树中重复出现
- 知识库不能同时属于两个空间文件夹

**当前差距：** Mock数据中存在：
- `kb-personal-3`（AI学习笔记）同时出现在 `default-personal-daily` 和 `default-personal-basics` 两个空间文件夹中
- `file-price-band`（价格带分析报告.docx, 属kb-public-4/团购预算池）被错误挂在 `folder-product`（商品资料）下
- `kb-personal-4`（客户素材归档）在 `fileTrees.personal` 中没有对应的TreeNode

**变更内容：**
- `default-personal-daily` 中去掉 `kb-personal-3`，仅保留在 `default-personal-basics`
- `file-price-band` 从 `folder-product` 移到 `folder-budget-pool`（属方案中心 kb-public-4）
- `fileTrees.personal` 中增加 `kb-personal-4` 客户素材归档 TreeNode及其子文档

### 需求3: 右键菜单规范

**PRD对照（§3.2.2.3 界面原型表第1行）：**「右键具有可管理权限的知识库 → 出现二级菜单，存在设置功能」。PRD要求不同节点类型展示不同菜单项。

**当前差距：** 文件树右键菜单未按节点类型区分：
- 根空间空白处：缺少「删除」选项
- 空间文件夹：使用 `openTreeBlankContextMenu`（id='__root__'），菜单显示「新建空间文件夹」「新建知识库」，不符合「新建知识库、设置、删除」的规范
- 知识库节点：缺少「设置」入口
- 知识库下普通文件夹：不应显示「设置」
- 具体文件：缺少「设置」

**变更内容：**
- 空间文件夹右键从 `openTreeBlankContextMenu` 改为 `openTreeContextMenu(folder.id, $event)`，新增 `contextSpaceFolder` computed 识别空间文件夹
- 右键菜单拆分为5个独立 `v-if/v-else-if` 分支：
  - 根空间空白处：新建空间文件夹、删除
  - 空间文件夹：新建知识库、设置、删除
  - 知识库(isKnowledgeBase=true)：新建文件夹、设置、删除
  - 知识库下普通文件夹：新建文件夹、删除
  - 具体文件：预览、设置、删除
- 保留「右键知识库行（KB列表）」和「右键文件行（文件列表）」的原有菜单

### 需求4: 根空间点击展示分组

**PRD对照（§3.2.3.1.1 普遍型/§3.2.3.1.5）：**「系统应在根空间优先展示空间文件夹，不直接展示知识库列表」——即点击"公共空间"/"个人空间"根节点时，主视图应展示文件夹分组列表，而不是直接跳到知识库列表。

**当前差距：** `switchSpace()` 后没有设置 `activeSpaceRoot` 状态，模板中也没有对应的「部门分组/知识分组」展示分支，点击根空间后直接展示知识库列表。

**变更内容：**
- 新增 `activeSpaceRoot = ref(false)` 状态变量
- `switchSpace()` 设置 `activeSpaceRoot = true`（点击根空间→显示分组）
- `openSpaceFolder()` 设置 `activeSpaceRoot = false`（进入空间文件夹→显示KB列表）
- `selectKb()` 设置 `activeSpaceRoot = false`（选择知识库）
- 模板新增 `v-else-if="!selectedKb && activeSpaceRoot"` 分支，展示 `defaultSpaceFolders` 分组表格（分组名/知识库数量/进入按钮）

### 需求5: 面包屑完整层级

**PRD对照（§3.1.2 层级结构）：** 五级结构「空间/空间文件夹/知识库/文件夹/文件」要求在面包屑中完整反映路径层级，每级可点击返回。

**当前差距：** 原面包屑仅展示「空间名/知识库名」或「空间文件夹名」，缺失中间文件树路径层级。例如"方案中心/方案中心案例库/团购方案"中的「团购方案」层级未能展示。

**变更内容：**
- 新增 `findTreeNodePath()` 辅助函数，在文件树中从根到活跃节点回溯完整路径
- 重写 `breadcrumbTrail` computed：先添加空间名+空间文件夹名，然后通过 `findTreeNodePath` 补全文件树中所有中间层级的节点标签
- 去重过滤：跳过已通过 `selectedKb` 或 `activeDefaultFolder` 展示过的节点
- 面包屑每级可点击调用对应导航函数（`switchSpace`/`openSpaceFolder`/`selectKb`）

### 需求6: 文件树拖拽排序

**PRD对照（§3.2.1.3 界面原型表第9行/§3.2.1.5）：** 「文件树、中间主view知识库支持拖拽移动位置」，「文件夹移动不得超出知识库，知识库移动不得超出空间文件夹」。

**当前差距：** 空间文件夹和收藏知识库卡片已有拖拽，但文件树中的文件/文件夹/知识库节点不支持拖拽。

**变更内容：**
- 新增 `draggingTreeNodeId`/`dragOverTreeNodeId` 状态
- 新增 `reorderTreeItems()`/`finishTreeDrag()`/`findNodeParent()` 函数
- 文件树每行节点div添加 `draggable` 属性 + `dragstart/dragover/drop/dragend` 事件
- 拖拽校验：仅允许同一父节点下排序（不超出知识库/空间文件夹范围），超出时Toast提示「只能在同一层级拖拽排序」

## 回收站界面

### 需求1: 关闭按钮位置与自动关闭

**PRD对照（§3.2.3.9.3 界面原型）：** 回收站应作为侧栏底部入口，关闭按钮应采用通用关闭交互，而非左箭头返回。

**当前差距：**
- 关闭按钮使用 `ChevronLeft` 左箭头（类似返回语义），位于标题左侧
- 点击文件树节点时不自动关闭回收站

**变更内容：**
- 移除 `ChevronLeft` 返回按钮，`X` 关闭按钮移到标题栏最右上角
- 搜索框移至标题栏右侧、关闭按钮左侧
- `closeRecycleIfOpen()` 添加到：
  - `toggleTreeNode()` — 主内容区文件夹/文件项点击时关闭
  - `switchSpace()` — 切换空间时关闭
  - `selectTreeNode()/selectKb()/previewFileFromTree()` — 已有此调用
- 回收站标题改为 `flex justify-between` 布局：左侧「回收站」标题，右侧「搜索框+关闭按钮」

> - 5级角色体系（OWNER/MANAGER/EDITOR/DOWNLOADER/READER）

> - 角色修改即时生效

> - 审计日志筛选与展示

## 知识库问答右侧边栏

### 需求1: 三栏布局联动

**PRD对照（§3.2.1.3 界面原型表格第6行）：** 预览时关闭小智问答 →「【左侧】文件树+【右侧】文件预览」的两栏布局。

**当前差距：** 当 `isThreeColumn`（文件预览+小智同时打开）时，预览内嵌在主内容区；关闭小智后，预览面板 `<aside>` 应重新出现占满右侧。当前布局机制已正确，无需额外代码变更。

**变更内容：** 无（逻辑已正确工作）

### 需求2: 会话记忆与持久化

**PRD对照（§3.2.4.1.3 界面原型表格）：** 「同一会话内连续对话——自动保留最近5轮对话作为上下文」，「消息自动保存（`kb_chat_history`），刷新页面或关闭侧栏后恢复」。

**当前差距：** 当前实现中 `qaMessages` 为纯前端内存保存，刷新页面后丢失。未实现 `kb_chat_history` 持久化存储。

**变更内容：**
- [二期] 对接 `POST /api/knowledge/qa/ask` 和 `GET /api/knowledge/qa/stream` 真实API
- [二期] 实现 `qaMessages` 的 localStorage/后端持久化
- [二期] 实现最近5轮上下文窗口管理（自动截断旧轮次）
