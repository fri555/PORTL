# AI工作门户 推广节奏

> 版本：v2.0  
> 日期：2026-06-25  
> 适用范围：AI中台第一期 MVP 种子用户试点  
> 核心结论：第一期不做全员推广，从方案中心种子用户开始，验证“首页提问 - 全屏工作台 - 方案中心主管 - 钉钉协同”的真实业务价值。

## 1. 推广目标

第一期推广目标不是让所有员工知道门户，而是让种子用户在真实方案沟通中用起来。

| 结果 | 判断标准 |
|------|----------|
| 有真实使用 | 种子用户用首页搜索或方案生成入口进入全屏工作台 |
| 有方案产出 | 方案中心主管能输出可讨论的一版方案 |
| 有协同动作 | 对话结果能沉淀为文档、待办或表格结构 |
| 有优化反馈 | 种子用户能指出方案质量、字段、知识、SKU 等缺口 |

## 2. 种子用户范围

| 用户/对象 | 首发价值 | 推广方式 |
|-----------|----------|----------|
| B2B线上团队 | 企业微信沟通中快速生成客户方案和回复话术 | 小范围账号开通 + 真实客户案例试用 |
| B2B线下团队 | 到店沟通时间短，需要快速方案初稿 | 现场演练 + 5分钟方案挑战 |
| 内部服务商 | 内部方案、组货、报价支持频繁 | 方案模板共建 |
| 外部服务商 | 面向外部客户交付，需要标准化口径 | 受控开放 + 输出审核 |
| 线下零售门店 | 低门槛收集客户需求和生成沟通内容 | 移动端轻量试用 |

## 3. 推广路径

### 3.1 第一圈：平台部与方案中心负责人

| 对象 | 重点 |
|------|------|
| 平台部/天团一号 | 确认一期范围、技术联调、能力边界 |
| 方案中心主管/业务负责人 | 确认模板、字段、案例池、输出口径 |
| 系统管理员 | 配置用户、角色、智能体、知识库、MCP 权限 |

### 3.2 第二圈：种子用户

| 动作 | 说明 |
|------|------|
| 开通账号 | 只给种子用户开方案中心主管权限 |
| 场景演练 | 用真实沟通场景测试需求澄清和方案初稿 |
| 结果复盘 | 记录哪里能用、哪里不准、缺什么知识和字段 |
| 快速迭代 | 每周更新提示词、模板、案例和工具联调状态 |

### 3.3 第三圈：扩展部门

当方案中心场景跑通后，再扩展到市场、电商、客服、人力、财务等部门，不在第一期全量宣发。

## 4. 推广节奏

### 第 0 周：准备

| 事项 | 输出 |
|------|------|
| 确认种子用户 | 名单、部门、权限 |
| 确认首发能力 | 首页快捷功能和方案中心主管能力 |
| 准备真实案例 | 线上/线下客户需求样例、成功方案样例 |
| 准备权限 | 普通用户、种子用户、运营管理员、系统管理员 |
| 准备联调清单 | 钉钉待办、钉钉文档、钉钉AI表格 |

### 第 1 周：首页与对话试用

| 动作 | 说明 |
|------|------|
| 10分钟上手 | 只讲首页搜索、快捷功能、全屏工作台 |
| 真实问题试问 | 每位种子用户至少输入 3 个真实任务 |
| 记录阻塞点 | 记录入口、权限、结果质量、工具调用问题 |

### 第 2 周：方案中心专项试用

| 动作 | 说明 |
|------|------|
| 5分钟方案挑战 | 输入客户需求，要求 5 分钟内得到方案初稿 |
| 字段补全测试 | 检查是否能追问预算、类目、场景、交付时间等必填项 |
| 案例引用测试 | 检查能否复用成功案例和通用模板 |
| 文案输出测试 | 输出沟通话术、营销文案和汇报提纲 |

### 第 3 周：复盘与迭代

| 复盘项 | 说明 |
|--------|------|
| 使用数据 | 登录人数、会话数、方案生成次数、快捷功能点击 |
| 质量数据 | 方案可用率、需要人工修改点、缺失字段 |
| 协同数据 | 待办、文档、表格生成或占位调用次数 |
| 缺口清单 | 知识库、案例池、SKU、PPT生成、上传能力等后续需求 |

## 5. 推广材料

### 面向种子用户

- 10分钟快速上手：从首页输入到全屏工作台
- 方案中心主管使用说明
- 好问题示例：如何描述客户、预算、类目和交付要求
- 结果修正示例：如何要求 AI 改方案、换口径、补字段

### 面向业务负责人

- 一期 MVP 范围说明
- 种子用户使用数据
- 方案质量复盘
- 知识库/案例池缺口
- 二期迭代建议

### 面向系统管理员

- 用户与角色配置说明
- 智能体、知识库、MCP 工具权限说明
- 普通用户隐藏后台与菜单规则
- 审计日志查看说明

## 6. 推广指标

| 指标 | 第一期目标 |
|------|------------|
| 种子用户 | 10-30 人 |
| 有效会话 | 100 次以上 |
| 方案生成 | 30 份以上 |
| 5分钟内出初稿 | 70% 以上场景达成 |
| 方案可继续讨论率 | 60% 以上 |
| 关键缺口收集 | 20 条以上 |
| 权限/入口阻塞 | 0 个 P0 问题遗留 |

## 7. 风险与应对

| 风险 | 表现 | 应对 |
|------|------|------|
| 范围再次发散 | 又想加需求池、市场、后台大屏 | 第一版只保留首页和全屏工作台 |
| 方案质量不稳定 | 输出泛泛、不贴业务 | 强化需求澄清、案例池和字段池 |
| SKU 不能及时接入 | 组货建议无法落地 | 第一版先输出文本策略，二期接 SKU 检索 |
| 钉钉能力联调不确定 | 待办/文档/表格调用失败 | 做降级：输出结构化内容 + 跳转/复制 |
| 普通用户误入后台 | 导致认知混乱或越权 | 菜单级、路由级、接口级三层权限控制 |

## 8. 推广口径

对上级：

> 第一期不是做完整 AI 门户，而是用最小范围验证 AI 中台能否在方案中心真实产生价值：一个入口、一个全屏工作台、一个方案中心主管、若干通用协同能力。

对种子用户：

> 你只需要从首页输入真实问题，系统会进入全屏工作台。先用它帮你澄清需求、生成方案初稿、整理文案和协同内容。

对项目组：

> 第一版少即是快。任何不能直接支撑方案中心种子用户试用的功能，都进入二期或三期。
