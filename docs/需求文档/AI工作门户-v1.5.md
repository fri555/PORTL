## 一、项目概述

### 1.1 背景痛点

集团内部已部署多个AI工具与系统，但存在以下问题：
- **入口分散**：员工不知道有哪些AI工具可用，需要四处寻找
- **缺乏运营**：AI使用停留在个人层面，没有形成组织级氛围
- **需求无闭环**：业务部门有AI需求无处提交，或提交后无追踪
- **培训缺失**：员工想用AI但不知从何入手
- **模型调用失控**：各部门各自接入大模型，无统一网关、无限额管控、无成本归集

### 1.2 产品核心定位

天马集团AI统一门户是集团AI能力的**唯一官方入口与运营阵地**，采用「高管决策服务 + 员工提效工具 + 开发者赋能」三轨设计：

- **向上**：为管理层提供行业情报、经营数据、全局AI进展视野
- **向下**：为全员提供AI工具聚合、需求闭环、学习成长平台
- **向开发者**：提供统一开发规范、能力接入、用量管理与智能体/Skill市场

门户**不重复建设业务系统**，核心承担四大职能：

| 职能 | 说明 |
|------|------|
| **统一入口** | 所有AI系统、智能体、工具的认证、访问、检索的唯一入口 |
| **标准规范** | 智能体开发规范、工具接入标准、API接口规范的制定与审核 |
| **数据聚合** | 模型用量、业务数据、运营指标的跨系统汇聚与可视化 |
| **氛围运营** | 资讯竞对情报驱动AI紧迫感，排名积分驱动参与度，培训驱动能力提升 |

### 1.3 核心设计原则

1. **轻量聚合**：现有AIGC、数字人、业务系统均以跳转/内嵌方式接入，不重复造轮子
2. **高度自定义**：首页组件支持自由拖拽、多尺寸一键切换，不同岗位可配置专属工作台
3. **分级授权**：基于统一身份体系实现功能、数据、应用的精细化权限管控，6类角色权责分离
4. **钉钉原生打通**：登录、消息、待办、培训全链路对接钉钉体系，降低使用门槛
5. **共建模式**：业务部门开发者可自主开发智能体/Skill，AI部门负责规范、审核与平台支撑
6. **模型网关自建**：门户作为集团统一模型网关，集中管理多厂商模型接入、限额、审计

### 1.4 术语定义

| 术语 | 定义 | 示例 |
|------|------|------|
| **AI工具** | 面向具体事务流程的开箱即用应用，底层由一个或多个Skill支撑 | AI商拍、智能文案、比价助手、文档翻译 |
| **智能体 (Agent)** | 垂类专精的AI专家，编排调度多个AI工具/Skill完成复杂任务，具备对话协作能力 | 「鞋类电商运营专家」编排比价、文案、商拍等工具 |
| **Skill** | 底层原子AI能力，不可再分的最小功能单元，供智能体和工具调用 | 商品描述生成、价格比对、图片背景替换 |
| **语义层** | 在知识库向量化基础上进行实体建模，构建部门/业务域的知识图谱，支持关联推理 | 商品实体→关联供应商→关联库存→关联销售数据 |
| **Chat BI** | 自然语言驱动的智能看板生成：一句话描述 → AI匹配数据源 → 自动选图 → 生成可视化看板 | 「生成6月各区域运动鞋销量对比」→ 自动出图 |
| **模型网关** | 门户自建的统一大模型调用入口，集中管理多厂商API密钥、限额、路由、审计 | 对接GPT/Claude/DeepSeek等多厂商 |

## 二、统一身份与角色权限体系

### 2.1 用户唯一主键绑定规则

以**钉钉用户ID**作为全系统唯一主键，与门户账号、大模型调用身份、任务体系、需求记录、学习数据、开发者身份一一绑定：

- 首次钉钉扫码登录自动创建门户账号，同步组织架构、部门、岗位信息
- 所有大模型调用、工具访问、操作行为均关联该唯一ID，实现用量精准统计、权限精准管控
- 跳转内部业务系统时携带统一身份Token，实现SSO单点登录，免二次认证
- 开发者身份与该ID绑定，其开发的智能体、工具、Skill自动关联个人账号

### 2.2 角色权限矩阵（6类独立角色 · 权责分离）

| 角色 | 身份说明 | 核心权限范围 |
|------|----------|--------------|
| **普通员工** | 集团全员默认角色 | 查看公开资讯、使用授权AI工具/智能体、提交需求、投票（每日3票）、查看个人模型用量与积分、学习课程、自定义个人首页 |
| **部门AI BP** | 各部门指定AI负责人 | 含普通员工全部权限 + 初审本部门需求、审核本领域资讯内容、查看本部门全量模型用量、管理本部门智能体/Skill权限、统筹本部门AI开发者 |
| **内容审核员** | AI运营团队专职内容岗 | 资讯终审、置顶/推荐设置、竞对情报管理、行业报告上传、培训内容维护、内容审核工作台操作 |
| **需求管理员** | AI项目组需求管理岗 | 需求终审、排期指派、进度管控、上线验收、需求数据统计、需求全流程状态流转管理 |
| **AI功能开发者** | 各部门技术/业务开发人员 | 智能体/Skill配置开发、工具接入申请、调用本应用API接口、查看所开发应用的模型用量、提交上线审核 |
| **系统管理员** | AI部门技术团队 | 全功能后台权限、工具/智能体/Skill上架审核、大模型配置与网关管理、角色权限分配、系统配置、全量数据查看 |

### 2.3 组织架构同步

- 钉钉组织架构自动同步至门户，支持手动调整
- 部门层级维护，支持部门合并、拆分后的数据归属调整
- AI BP 人员由系统管理员指定，与钉钉部门负责人可以不是同一人

## 三、前台功能模块

### 3.1 首页：iTab 式 AI 桌面工作区

#### 3.1.1 编辑权限与默认配置

- **全员可编辑**：所有用户均可自定义自己的首页布局、组件、尺寸
- **角色默认模板**：新用户首次登录时，按角色加载对应的默认首页配置
- **模板更新策略**：管理员更新某角色默认模板后，仅影响新用户；已自定义过的老用户不受影响
- **一键重置**：用户可将自己的首页重置为当前角色的默认模板
- **桌面化体验**：首页作为用户进入 AI 能力的第一屏，不采用传统信息门户堆叠，而采用类似 iTab/手机桌面的工作区形态，用户通过搜索框和桌面组件完成主要操作

#### 3.1.2 栅格与组件规格

PC 端采用 **14×4 自由栅格布局**，组件像手机桌面图标与小组件一样自动贴合排列；移动端首页不展示复杂组件网格，仅保留超级搜索框与底部 Tab，降低小屏认知负担。栅格支持自由拖拽排序，相邻组件在拖动或尺寸调整后自动适应、补位、顺延。

用户可切换三种桌面密度：

| 密度 | 说明 | 适用场景 |
|------|------|----------|
| 宽松 | 12列×2行 | 大屏展示、偏视觉浏览 |
| 标准 | 14列×4行 | 日常办公推荐 |
| 紧凑 | 16列×5行 | 高频工具用户、组件数量较多 |

提供7种预设标准尺寸，尺寸名称即真实栅格占位：

| 组件规格 | 栅格占位 | 适用场景 |
|----------|----------|----------|
| 1×1 | 占1个桌面单元 | 单个工具入口、个人用量速览、单一数据指标 |
| 1×2 | 占1列×2行 | 模型用量、窄列表、竖向状态组件 |
| 2×1 | 占2列×1行 | 小型快捷入口、横向单功能组件 |
| 2×2 | 占2列×2行 | 待办、小型榜单、小型信息卡 |
| 2×4 | 占2列×4行 | 竖向长组件、纵向列表 |
| 4×2 | 占4列×2行 | 资讯热榜、竞对速览、部门榜单、中等看板预览 |
| 4×4 | 占4列×4行 | 经营数据、重点业务看板、首屏核心组件 |

**尺寸切换交互**：不再要求用户进入编辑模式。用户可直接拖动组件调整位置；右键点击组件弹出上下文菜单，支持调整尺寸、前移、后移、删除等操作。切换后 14×4 栅格自动重排，超出区域自动顺延到下一行。

**拖动排序交互**：组件采用坐标模型保存位置，每个组件持有 x/y 坐标与尺寸占位。用户可将组件拖动到任意空白格或目标格，组件之间允许保留空白，不强制自动紧凑补位。拖动过程中桌面底层展示网格参考线，释放后提交坐标位置；如发生碰撞，仅对冲突组件做局部避让，不进行全局强制重排。虚线网格必须使用与真实组件布局一致的列数、行高和间距生成，不允许用独立背景纹理近似模拟。该交互必须遵循尼尔森十大交互原则：拖动时提供明确反馈，状态可见，避免组件突然丢失。

**1×1 图标组件**：1×1 组件不使用外层卡片，应以桌面图标形态直接展示。支持配置文本、图标颜色、跳转链接；暂不要求上传自定义图片。

**不同尺寸差异化展示**：组件切换尺寸时不做简单等比缩放，而是按尺寸切换信息架构。例如经营数据在小尺寸展示核心指标，在 4×4 展示完整指标矩阵；竞对快照、行业头条、部门AI价值榜等列表类组件必须保持纵向单列阅读，一行一条信息，宽尺寸只增加可见条数、摘要长度或元信息，不能改成横向多列导致信息截断；工具类组件在 1×1 展示单图标入口，在 2×1 展示紧凑入口，在 2×2 展示工具网格。

**坏尺寸规避**：如某组件在特定尺寸下无法保证信息完整与视觉稳定，应从允许尺寸中移除，不允许用户调出明显截断、拥挤、不可读的坏状态。尺寸菜单只展示当前组件支持的尺寸，不展示禁用项。例如模型用量不支持 1×1、1×2、2×1 小尺寸，仅保留 2×2 与 4×2 数据看板态；数字人平台不支持 2×1 横条态。

**菜单层级与可见性要求**：组件右键菜单、尺寸菜单、组件选择弹窗必须使用全局浮层层级，不能受卡片、网格、滚动容器或下方组件遮挡。尺寸菜单必须根据视口位置自动上移或启用内部滚动，确保底部组件也能完整选择尺寸。该交互属于基础 QA 必测项。

**一屏展示原则**：默认员工首页按标准 14×4 栅格完整填满首屏（共56格），避免出现明显空白和竖向滚动条。默认布局采用 4×4 核心数据组件、4×2 信息组件、2×2 任务组件、1×2 状态组件、2×1 横向入口和 1×1 图标入口组合；用户新增组件后如超出首屏，可通过右键桌面菜单切换紧凑布局或手动删除/调整组件。

模式切换要求：
- 当前模式必须有明确的选中态，符合“系统状态可见”原则
- 推荐内容必须跟随模式即时变化，符合“识别优于回忆”原则
- MVP 首页的「今日推荐」组件也必须跟随当前搜索模式同步变化，避免搜索框与下方组件表达割裂。
- 任务助理、AI生图等模式的推荐内容采用浮层/毛玻璃面板覆盖在组件区上方，不因模式切换挤压下方桌面布局
- 发送、附件、语音等高频动作位置保持稳定，降低学习成本
- 点击语音按钮必须有明确反馈，例如按钮高亮与「语音输入中」状态提示
- 未输入内容时点击发送，可自动填入该模式下的默认示例，避免空操作

#### 3.1.7 金币/点数与 Token 消耗口径

门户不采用“限制次数”的产品表达，而采用金币/点数与 Token 挂钩的统一消耗口径：

1. **统一余额**：用户侧展示金币/点数余额，避免直接暴露复杂模型 Token 差异
2. **模型倍率**：不同模型、不同任务按倍率换算，例如标准文本模型为 1.0，高阶推理/图像/视频模型按更高倍率计费
3. **执行前预估**：提交任务前展示预估金币消耗区间，执行后按实际 Token 和任务类型结算
4. **组织归集**：个人消耗可归集到部门、项目、智能体、Skill，服务后续成本核算
5. **异常保护**：当预估消耗超过个人或部门阈值时，提示确认或走审批；不是简单阻断次数

### 3.2 智能工作台（工作台 + 能力市场模式）

> 智能工作台不再默认进入单一智能体市场，而是先提供“能做事”的工作区：用户在上方完成对话、创建和调度，下方发现专家、专家团、智能体、MCP能力与技能。

#### 3.2.1 页面布局

- **顶部工作台区**：展示当前用户的 AI 能力编排入口，包含对话式工作台、快捷创建入口、个人能力资产概览。
- **对话式工作台**：保留一个可直接输入的对话框，用户可以选择专家/智能体进行协作，也可以用自然语言描述“我要创建什么能力”“我要设置什么定时任务”。对话区需支持附件入口、快捷提示词、发送反馈和后续流式回复扩展。
- **全屏对话工作台**：首页超级搜索框和工作台对话框发送任务后，进入全屏对话工作台承接后续交互。全屏工作台参考主流 AI 对话产品形态，左侧为历史会话/任务记录，中间为当前对话主区，右侧为当前智能体、知识库、技能、定时任务等能力上下文。
- **全屏布局约束**：全屏工作台应占满浏览器可用工作区，主页面不产生整体滚动；左侧历史栏、中间消息区、右侧上下文栏分别在自身容器内滚动。左右侧栏均支持折叠，折叠后保留窄图标列，中间主对话区自动扩展。
- **返回规则**：全屏对话页不再单独放置“返回工作台”按钮，避免打断对话视觉；回首页或回工作台由全局品牌入口、浏览器返回或外层导航负责。
- **对话动效**：从首页/工作台进入全屏对话、左右栏折叠/展开、历史会话切换时使用 150-300ms 轻量过渡，避免夸张动效影响办公效率。
- **快捷创建区**：提供「创建智能体」「创建知识库」「创建技能」「定时任务」四类高频入口。点击后使用弹窗表单收集名称、目标说明、知识库/权限/执行时间等关键字段，先形成草稿或任务配置，再进入详细编辑。
- **能力资产概览**：展示我的智能体、知识库、已发布技能、定时任务数量，点击进入对应管理页，符合“识别优于回忆”和“用户控制与自由”原则。
- **下方能力市场**：采用标签筛选结构，分类包括「专家、专家团、智能体、MCP能力、技能」。每张卡片展示名称、适用场景、来源/部门、评分或可用状态、查看/加入入口。
- **详细管理页保留**：智能体市场、技能商店、知识库、定时任务、模型用量仍保留独立二级页面，用于复杂筛选、审核、编辑和日志查看；工作台首页负责聚合入口与快速启动。
- **响应式要求**：PC端两栏展示对话工作台与能力资产；移动端改为单列堆叠，保留创建入口、对话框和能力市场标签，不出现横向溢出。

#### 3.2.2 智能体市场

1. **顶部筛选区**：业务分类标签（营销内容、商品供应链、零售运营、客户服务、财务行政、技术研发）+ 关键词搜索 + 我的收藏/我的开发筛选
2. **我的常用区**：横向排列已收藏的智能体头像，点击快速唤起对话
3. **智能体卡片网格**：
   - 卡片信息：智能体形象图标、名称、能力标签（如「鞋类电商运营专家」）、能力等级、开发者、一句话功能描述、已编排工具/Skill数量
   - 操作按钮：「收藏」「开始协作」；开发者对自己创建的智能体显示「编辑」
   - 点击「开始协作」从右侧弹出对话抽屉，在当前页完成交互，支持上下文记忆、附件上传
4. **一句话生成智能体**：
   - 开发者入口：输入自然语言描述期望的智能体能力，AI自动生成完整智能体骨架
   - AI自动完成：智能体名称、形象图标、能力描述、提示词、关联工具/Skill编排、知识库绑定
   - 生成后进入编辑页面，开发者逐项确认和修改，完成后提交审核
   - AI部门审核通过后上架到市场
5. **开发者能力**：支持可视化配置智能体——设置提示词、绑定底层模型、关联知识库、编排工具/Skill、配置欢迎语与快捷指令

#### 3.2.3 Skill市场

1. **Skill定位**：底层原子AI能力，不可再分，供智能体和AI工具调用。与智能体市场同级展示
2. **卡片信息**：Skill名称、能力图标、所属业务分类、API端点、调用格式、开发者、支持的模型
3. **一句话生成Skill**：
   - 输入自然语言描述期望的Skill功能，AI自动生成Skill定义
   - 自动生成：输入/输出Schema、Prompt模板、推荐模型配置
   - 生成后进入编辑确认页，开发者确认后提交审核
4. **Skill调用**：开发者可查看Skill的API文档、请求示例，申请调用权限

#### 3.2.4 我的任务（含定时任务）

整合即时任务与定时任务，支持钉钉待办与机器人提醒联动：

1. **顶部标签切换**：即时任务 / 定时任务
2. **任务列表**：展示任务名称、关联智能体、执行状态、创建/上次运行时间、操作按钮（查看、编辑、手动触发、启停）
3. **定时任务配置项**（弹窗表单）：
   - **基础信息**：任务名称、任务描述、选择关联智能体
   - **触发设置**：
     - 自然语言输入（如「每周一9点执行」），前端调用大模型翻译为Cron表达式，**结果展示给用户确认**后才生效
     - Cron表达式精确配置（高级模式）
     - 生效时间范围设置
   - **执行内容**：配置智能体执行指令、关联数据源/知识库、输出格式要求
   - **推送设置**：
     - 钉钉个人待办：执行结果生成待办事项，推送到发起人钉钉待办列表
     - 钉钉群机器人：推送结果到指定钉钉群
     - 站内信通知
   - **异常规则**：失败自动重试次数、失败告警推送
4. **任务日志**：点击任务查看历史执行记录、输出结果、报错信息
5. **积分消耗机制**（定时任务并发控制）：
   - 定时任务每次执行消耗积分，消耗量与关联模型预估Token消耗挂钩（1M token ≈ 1积分）
   - 积分不足时任务自动暂停，充值后恢复
   - 不同优先级任务消耗系数不同：普通任务 ×1.0，高优先级 ×1.5

#### 3.2.5 知识库管理

按业务主题分类的企业知识库，支持绑定智能体实现专属问答：

1. **左侧分类树**：按部门/业务域展示知识库目录（商品知识库、门店运营库、财务制度库等）
2. **右侧文档列表**：展示文档名称、格式、更新时间、操作（预览、编辑、删除）
3. **核心功能**：
   - 批量上传PDF/Word/Excel/PPT/TXT，自动解析向量化存储
   - 方案中心知识库首批支持组货逻辑、成功案例池、通用预算池、个性化方案池、字段定义、SPU/SKU待查模板、PPT/PDF交付模板入库
   - 权限设置：全集团可见、指定部门可见、指定人员可见
   - 召回规则配置：相似度阈值、召回条数、引用展示规则
   - 绑定到指定智能体，配置后智能体可调用知识库内容作答
   - 智能体每次回答必须返回引用文件/片段摘要；前端在右侧上下文展示引用来源，便于业务人员判断方案依据
   - 当智能体无法回答或置信度不足时，自动记录“知识缺口”，包含问题、会话、原因、责任人和处理状态，供方案中心后续补充完善
4. **语义层建设**（知识库上层）：
   - 在向量化知识库基础上进行**实体建模**：自动抽取文档中的核心实体（如商品、供应商、门店、指标等）
   - 构建**网状知识图谱**：实体间关系自动发现与人工标注结合
   - 支持**关联推理查询**：如「查询A商品→关联B供应商→关联C库存数据」
   - 智能体调用知识库时可基于语义层进行多跳推理，而非仅关键词召回

#### 3.2.6 大模型用量统计

按权限范围展示对应层级的模型用量数据：

1. **顶部筛选栏**：时间范围（日/周/月/自定义）、模型类型筛选、维度切换（个人/部门/我的应用）
2. **核心指标卡**：总调用次数、消耗Token量、预估费用、积分消耗、人均调用量
3. **趋势图表**：对应时间范围内的调用量、费用趋势、积分消耗折线图
4. **分级数据维度**：

| 角色 | 可见范围 |
|------|----------|

- **右侧边栏**：三榜排名（使用者排名、提需求排名、意见领袖排名）

#### 3.6.2 需求全流程流转

```
提交需求 → 部门AI BP初审 → 需求管理员评审 → 已排期 → 开发中 → 测试验收 → 已上线 → 已完成
    ↓              ↓              ↓                                    ↑
  退回修改       退回修改        退回修改                            验收不通过
```

| 流转节点 | 负责角色 | 核心工作 |
|----------|----------|----------|
| **提交需求** | 全员 | 填写需求表单，描述痛点与期望效果，支持图片+AI图片理解辅助填写 |
| **部门初审** | 提出部门AI BP | 审核需求合理性、业务价值，确认是否本部门可承接开发 |
| **需求评审** | 专职需求管理员 | 评估技术可行性、是否纳入统一规划，判定开发模式，设置排期优先级 |
| **开发执行** | 业务部门AI开发者/AI部门 | 业务部门可自主开发，AI部门提供技术支撑与规范指导 |
| **测试验收** | 提出人+需求管理员 | 功能验收、上线前合规审核 |
| **上线闭环** | 系统管理员/需求管理员 | 正式上线，更新状态，通知提案人与投票人 |

#### 3.6.3 核心功能

1. **极简提需求（AI图片理解辅助）**
   - 轻量化表单：需求标题、痛点描述、期望效果、紧急程度、附件上传
   - **图片上传+AI理解**：上传需求截图、原型草图、问题截图；AI自动识别图片内容，提取功能点、问题描述、场景信息，自动填充到需求描述字段，用户可修改补充
   - AI辅助填写：输入一句话描述，AI自动补全需求详情、优化表述
   - 重复检测：提交时自动匹配相似需求，引导先投票避免重复
2. **需求投票**
   - 每人每日3票，投票数影响优先级排序
   - 需求状态变更时，自动通知所有投票用户
3. **需求追踪**
   - 可视化状态时间线：每个节点处理人、处理时间、备注
   - 支持评论补充，在线沟通细节、上传进度文件
4. **我的需求**：我提交的 + 我投票的，统一管理查看进度
5. **排名体系**（需求池页面展示）：
   - 使用者排名：按AI工具使用频次/时长排名
   - 提需求排名：按提交需求数+投票权重排名
   - 意见领袖排名：按投稿/评论/被点赞综合排名
   - 部门排名：按部门维度聚合排名
   - 排名周期：日榜/周榜/月榜
   - 升降标识：排名变化可视化展示

## 五、大模型统一网关

> 门户自建，作为集团统一模型调用基础设施。所有业务系统、智能体、AI工具的模型调用均经此网关。

### 5.1 多厂商模型接入

- 支持接入多家大模型厂商：OpenAI (GPT系列)、Anthropic (Claude系列)、DeepSeek、智谱 (GLM系列)、百度 (文心系列)、阿里 (通义系列) 等
- 统一API密钥管理：每个厂商独立配置API密钥，支持密钥轮换
- 接口适配层：将不同厂商的API差异封装为统一调用接口，上层应用无需感知厂商差异
- 模型能力标签：每个模型标注能力属性（对话/代码/推理/多模态/Embedding等）

### 5.2 调用限额体系

| 层级 | 限额类型 | 配置方式 |
|------|----------|----------|
| 全集团 | 月度总Token限额、月度总费用限额 | 系统管理员配置 |
| 部门 | 月度Token限额、月度费用限额 | 按部门分配，超额告警 |
| 个人用户 | 月度Token限额、积分兑换额度 | 默认值 + 按角色差异化 |
| 开发者应用 | 单个智能体/Skill/工具的独立Token限额 | 开发者申请 + 管理员审批 |

- 超额策略：先告警，超限后返回降级模型或拒绝调用
- 积分兑换：用户可消耗积分兑换额外Token额度（1积分 ≈ 1M token）

### 5.3 模型路由

- **默认路由**：智能体/工具创建时指定默认模型
- **降级策略**：主模型不可用时自动切换备用模型（如 GPT-4 → Claude Sonnet → DeepSeek）
- **成本路由**：低优先级任务可路由到低成本模型
- **多模态路由**：根据请求内容自动选择支持对应模态的模型

### 5.4 用量与成本管理

- 实时用量仪表盘：全集团/分部门/分模型/分应用的调用量、Token消耗、费用
- 成本分摊：按部门的实际调用量自动计算分摊费用
- 费用预估：基于历史趋势预估当月/下月费用
- 异常告警：单用户/单应用调用量异常飙升自动告警

### 5.5 调用审计

- 全量调用日志：时间、用户、调用方应用/智能体、使用模型、输入Token、输出Token、费用
- 敏感内容审计：自动检测输入/输出中的敏感内容，记录并告警
- 日志保留策略：原始日志保留90天，聚合数据永久保留

## 六、积分与排名体系

### 6.1 积分模型

#### 6.1.1 积分获取

| 行为 | 积分 |
|------|------|
| 每日登录 | +1 |
| 使用AI工具/智能体（每次） | +1 |
| 提交需求（经初审通过） | +5 |
| 需求被采纳上线 | +20 |
| 发表技术分享投稿（经审核发布） | +10 |
| 完成课程学习 | +5 |
| 评论/点赞（每日上限5次） | +1 |
| 智能体/Skill上架 | +30 |

#### 6.1.2 积分消耗

| 行为 | 消耗 |
|------|------|
| 大模型调用额度兑换 | 1积分 = 1M token |
| 定时任务执行（每次） | 按关联模型预估Token消耗换算（1M token ≈ 1积分） |
| 高优先级定时任务 | ×1.5 系数 |

#### 6.1.3 金币绑定

- 积分与公司内部金币制度互通：积分可按比例兑换为金币
- 兑换比例由系统管理员配置，默认可配置
- 金币可用于公司内部福利兑换（具体规则由HR制定，门户仅提供积分→金币的兑换接口）

### 6.2 三榜排名体系

#### 6.2.1 排名维度

| 排名 | 计算维度 | 首页展示 |
|------|----------|----------|
| **使用者排名** | AI工具使用频次 + 时长 + Token消耗量 | TOP5 |
| **提需求排名** | 提交需求数 + 被采纳数 + 投票权重 | TOP5 |
| **意见领袖排名** | 投稿数 + 评论数 + 被点赞数 + 课程完成数 | TOP5 |
| **部门价值排名** | AI渗透率 + 提效数据 + 需求闭环率 + 部门Token消耗 | TOP10 |

#### 6.2.2 排名机制

- 日榜/周榜/月榜三级，首页默认展示月榜
- 升降标识：排名变化 ±N 可视化展示
- 积分体系作为排名基础数据源
- 每日积分获取上限：10分（防刷）
- 异常检测：异常行为自动标记，人工复核

#### 6.2.3 首页展示

需求池页面右侧边栏展示三榜排名。首页可选配「部门AI价值榜」组件。

## 七、全链路钉钉集成方案

### 7.1 身份认证与组织同步

- 钉钉扫码登录，免账号密码，实现SSO单点登录
- 自动同步钉钉组织架构、人员信息、部门关系，实时更新
- 跳转内部业务系统自动携带身份Token，免二次登录

### 7.2 全场景消息推送

| 场景 | 推送内容 | 推送渠道 |
|------|----------|----------|
| 资讯推送 | 订阅分类更新、重磅头条 | 钉钉工作消息 |
| 需求通知 | 状态变更、审核结果、上线通知 | 钉钉工作消息 |
| 任务通知 | 定时任务执行结果、失败告警 | 钉钉工作消息 + 待办 |
| 培训通知 | 课程学习提醒、完成通知 | 钉钉工作消息 |
| 系统通知 | 账号安全、系统维护公告 | 钉钉工作消息 |
| 开发者通知 | 应用审核结果、调用超限告警 | 钉钉工作消息 |

### 7.3 定时任务与钉钉待办联动

- 定时任务执行结果可生成钉钉个人待办，同步到用户钉钉待办列表
- 支持配置钉钉群机器人，任务结果自动推送到指定部门群
- 重要告警信息可触发钉钉DING提醒

### 7.4 培训体系打通

- 对接钉钉云课堂，课程信息、学习进度、完成状态双向同步
- 学习提醒、课程通知通过钉钉工作消息推送
- 学习积分同步到门户积分体系

### 7.5 移动端钉钉工作台入口

- 钉钉工作台放置门户入口图标，点击跳转移动端H5
- 钉钉内免登直接访问
- 仅做入口链接跳转，不做深度JSAPI内嵌
