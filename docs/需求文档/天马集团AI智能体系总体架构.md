## 1. 项目背景与目标

### 1.1 集团业务背景

南京天马网络科技集团是运动鞋服赛道的复合型企业，具备三重身份：

1. **运动鞋服电商品牌方**：运营自有/代理运动鞋服品牌，覆盖天猫、京东、抖音、得物、拼多多等多平台，在营 SKU 数千个。
2. **运动鞋服供应链/产业带平台企业**：连接上游工厂与下游中小商家，提供货源、趋势、选品服务。
3. **自营云仓**：自有仓储履约体系，掌握真实库存、动销、周转数据。

集团已部署多个AI工具与系统，但面临**入口分散、缺乏运营、需求无闭环、培训缺失**等问题。同时各AI系统技术栈不统一（门户用 Vue3+FastAPI，内容工厂用 React+Node+Python），缺乏顶层架构设计串联。

### 1.2 建设目标

构建天马集团统一的AI智能体系，实现：

1. **统一架构**：5个子系统+共享基础设施的分层平台架构，技术栈规范化
2. **统一入口**：AI门户作为全员唯一入口，SSO全体系通行
3. **能力复用**：通用智能体平台作为底层AI引擎，三方集成作为统一集成总线，避免各子系统重复建设
4. **数据贯通**：统一数据层+消息总线，子系统间数据流动与协作（如库存×内容联动）
5. **分步建设**：按依赖关系分阶段交付，降低风险

### 2.2 核心设计原则

| 原则 | 说明 |
|------|------|
| **分层依赖** | 上层只能调用下层，不可反向依赖。第3层业务应用不直接互调，需通过第2层或消息总线 |
| **能力供给** | 第2层是"能力层"——智能体平台供AI能力，三方集成供连接能力，所有业务应用共享 |
| **一次建设复用** | 第1层基础设施全体系共享，避免每个子系统重复造轮子（SSO、存储、消息） |
| **统一入口** | 所有面向用户的AI能力最终通过第4层门户触达员工，门户是唯一入口 |
| **技术栈灵活** | 各层内部子系统可选技术栈，层间通过API契约+消息协议协作 |
| **数据流动** | 数据左→右流动（采集→分析→生成→决策），效果数据回灌形成飞轮 |

### 2.3 子系统关系矩阵

| 调用方↓ \ 被调方→ | 门户 | 内容工厂 | 供应链 | 智能体平台 | 三方集成 | 基础设施 |
|---|---|---|---|---|---|---|
| **AI门户** | — | 嵌入/跳转 | 嵌入/跳转 | 调用(AI助手) | 调用(工具跳转) | 全部依赖 |
| **内容工厂** | 提需求/资讯 | — | 调用(库存数据) | 调用(Agent编排) | 调用(模型/电商API) | 全部依赖 |
| **供应链** | 提需求/资讯 | 调用(内容需求) | — | 调用(预测Agent) | 调用(ERP/云仓API) | 全部依赖 |
| **智能体平台** | — | — | — | — | 调用(模型API) | 存储/消息依赖 |
| **三方集成** | — | — | — | — | — | 存储/消息依赖 |

**关键协作关系**：
- 内容工厂 ↔ 供应链：库存×内容联动（畅销款加码内容、滞销款清仓内容）
- 内容工厂 → 三方集成：调用AI模型API生成内容、调用电商API分发内容
- 门户 → 所有业务应用：SSO跳转、页面嵌入、AI助手调用智能体平台
- 所有业务应用 → 智能体平台：调用Agent/工作流/RAG能力

## 3. 第1层 · 共享基础设施层

这是全体系共享的技术底座，一次建设、各层复用。避免每个子系统重复搭建SSO、存储、消息等。

### 3.1 统一身份与权限

```
┌─────────────────────────────────────────────────────┐
│  统一身份认证中心 (IAM)                               │
│                                                      │
│  ┌───────────┐  ┌───────────┐  ┌───────────┐       │
│  │ 认证协议   │  │ Token管理  │  │ 组织架构   │       │
│  │ OAuth2     │  │ JWT       │  │ 部门/层级  │       │
│  │ CAS/SAML  │  │ Access 2h │  │ 同步AD/LDAP│       │
│  │ LDAP      │  │ Refresh7d │  │ 定时同步   │       │
│  └───────────┘  └───────────┘  └───────────┘       │
│                                                      │
│  ┌───────────┐  ┌───────────┐  ┌───────────┐       │
│  │ RBAC权限   │  │ 统一用户表  │  │ 审计日志   │       │
│  │ 角色/权限  │  │ 全体系共享  │  │ 登录/操作  │       │
│  │ 资源管控   │  │ 首次登录建号│  │ 安全审计   │       │
│  └───────────┘  └───────────┘  └───────────┘       │
└─────────────────────────────────────────────────────┘
         │
         ▼ SSO Token 全体系通行
  ┌──────┬──────┬──────┬──────┐
  │ 门户  │内容工厂│ 供应链│智能体│  各子系统校验JWT即可
  └──────┴──────┴──────┴──────┘
```

| 能力 | 说明 | 技术选型 |
|------|------|---------|
| 统一登录 | 一次登录、全体系通行，门户与各子系统共享会话 | OAuth2 + JWT，对接集团现有CAS/LDAP |
| 统一用户表 | 全体系共用一张用户表，含部门/角色/权限 | PostgreSQL `users` + `departments` |
| RBAC权限模型 | 统一角色定义（user/editor/admin等），各子系统按需扩展业务权限 | 角色表 + 权限表 + 资源表 |
| 组织架构同步 | 定时从AD/LDAP同步组织架构，保持与集团HR系统一致 | Celery定时任务 + LDAP适配器 |
| 审计日志 | 统一记录登录、关键操作，供安全审计 | 结构化日志 → ELK |

**与现有文档的对齐**：AI门户已有的SSO方案（OAuth2+JWT、Access 2h/Refresh 7d、首次登录建号）升级为全体系共享的IAM，各子系统不再各自实现登录。

|--------|------|--------|

### 3.3 消息总线

```
┌──────────┐                    ┌──────────────┐
│ 内容工厂  │─── 内容生成完成 ──→│              │
└──────────┘                    │              │──→ 供应链: 触发库存×内容联动
┌──────────┐                    │  消息总线     │
│ 供应链    │─── 库存预警 ─────→│ Kafka /      │──→ 内容工厂: 触发清仓内容生成
└──────────┘                    │ RabbitMQ     │──→ 门户: 推送通知给相关人员
┌──────────┐                    │              │
│ 三方集成  │─── 效果数据回采 ──→│              │──→ 内容工厂: 效果回灌预测模型
└──────────┘                    └──────────────┘
```

**职责**：子系统间异步解耦通信。关键事件流：
- `content.generated` — 内容工厂生成完成 → 供应链(库存联动) + 门户(动态更新)
- `inventory.alert` — 供应链库存预警 → 内容工厂(清仓/加码内容) + 门户(通知)
- `effect.data.synced` — 三方集成回采效果数据 → 内容工厂(效果回灌)
- `demand.status_changed` — 门户需求状态变更 → 通知提出者/投票者

**选型**：一期用 RabbitMQ（轻量、易部署、够用）；规模扩大后可升级 Kafka（高吞吐、流处理）。

### 3.4 对象存储与监控

| 能力 | 说明 | 技术选型 |
|------|------|---------|
| 对象存储 | 图片/视频/文件统一存储，全体系共享桶（按子系统分prefix） | MinIO（自建）+ 阿里云OSS（备份/CDN） |

## 4. 第2层 · AI能力层

这一层是"能力供给层"，包含两个子系统：**通用智能体平台**（供AI能力）和**三方平台集成**（供连接能力）。所有第3层业务应用都通过它们获取AI能力和外部连接。


**联动决策矩阵**（天马独有差异化，内容工厂已有设计，供应链平台补充库存侧）：

| 库存状态 | 内容状态 | 联动决策 |
|---------|---------|---------|
| 畅销 + 库存充足 | 内容齐全 | 内容加码（追加视频/投放素材） |
| 畅销 + 库存告急 | — | 内容维稳 + 供应链补货建议 |
| 滞销 + 库存高 | — | 清仓内容（促销视频/特价文案）+ 供应链呆滞预警 |
| 滞销 + 库存低 | — | 内容止损（停止投入） |
| 新品 + 起爆预测高 | 无内容 | 优先排期全套内容生成 + 供应链预备货 |

## 7. 跨层数据流与技术栈规范

### 7.1 跨层通信规范

层间通信有三种模式，按场景选择：

```
┌─────────────────────────────────────────────────────────────────┐
│  通信模式选择决策树                                               │
│                                                                  │
│  需要实时返回结果？                                               │
│    ├─ 是 → 同步API调用 (REST/gRPC)                              │
│    │       适用: Agent执行、工作流启动、RAG检索、模型调用         │
│    │       规范: 统一 /api/v1/ 前缀, JWT认证, 统一响应格式        │
│    │                                                             │
│    └─ 否 → 需要可靠投递？                                        │
│           ├─ 是 → 消息总线 (RabbitMQ)                           │
│           │       适用: 跨子系统事件通知、状态变更、异步任务       │
│           │       规范: 事件名 {domain}.{action} 格式             │
│           │                                                     │
│           └─ 否 → 数据同步？                                     │
│                  ├─ 是 → CDC/定时同步                             │
│                  │       适用: 主数据同步、组织架构同步            │
│                  │       规范: 通过三方集成层适配器               │
│                  │                                               │
│                  └─ 否 → 共享存储                                 │
│                          适用: 文件/素材、分析数据                │
│                          规范: 对象存储按prefix隔离              │
└─────────────────────────────────────────────────────────────────┘
```

### 7.2 统一API规范

| 规范项 | 规则 |
|--------|------|
| 路径前缀 | 每个子系统统一 `/api/v1/` 前缀 |
| 认证 | Bearer JWT（由第1层IAM签发，全体系通用） |
| 响应格式 | `{ "code": 0, "message": "success", "data": {}, "pagination": {} }` |
| 错误码 | 0成功 / 40000-40099参数 / 40100-40199认证 / 40300-40399权限 / 40400-40499不存在 / 50000-50099服务器 |
| 分页 | `?page=1&page_size=20`，响应含 `total`/`total_pages` |
| 幂等 | 写操作支持 `Idempotency-Key` 头，防止重复提交 |
| 限流 | 按子系统/用户/API维度限流，超限返回429 |
| 文档 | 各子系统提供 OpenAPI 3.0 规范，门户聚合展示 |

### 7.3 事件总线规范

**事件命名**: `{domain}.{action}`
- domain: `content`(内容工厂) / `supply`(供应链) / `portal`(门户) / `agent`(智能体) / `integration`(三方集成)
- action: `created` / `updated` / `deleted` / `generated` / `alert` / `synced` / `changed`

**事件消息格式**:

```json
{
  "event_id": "uuid",
  "event_type": "content.generated",
  "timestamp": "2026-06-21T10:00:00Z",
  "source": "content-factory",
  "data": {
    "asset_id": "uuid",
    "sku_id": "SKU001",
    "scenario": "commercial_shoot",
    "status": "completed"
  },
  "trace_id": "uuid"
}
```

**核心事件清单**:

| 事件 | 发布者 | 订阅者 |
|------|--------|--------|
| `content.generated` | 内容工厂 | 供应链(库存联动)、门户(动态更新) |
| `inventory.alert` | 供应链 | 内容工厂(清仓/加码)、门户(通知) |
| `demand.forecast` | 供应链/内容工厂 | 对方(备货/预生产) |
| `effect.data.synced` | 三方集成 | 内容工厂(效果回灌) |
| `demand.status_changed` | 门户 | 提出者/投票者(通知) |
| `model.invoke.logged` | 三方集成 | 门户(成本看板) |
| `user.points_changed` | 门户 | 门户(排名更新) |

### 7.4 技术栈规范

| 层 | 组件 | 技术栈 | 统一要求 |
|----|------|--------|---------|
| **第4层·入口** | AI门户前端 | Vue3 + TypeScript + TailwindCSS + Vite | **统一**（门户是唯一入口，必须统一） |
| | AI门户后端 | Python + FastAPI | **统一** |
| **第3层·业务** | 内容工厂前端 | React + TypeScript（已有） | 允许（已有代码） |
| | 内容工厂后端 | Python（已有） | **统一用Python** |
| | 供应链前端 | 待定（建议Vue3或React） | 允许自选 |
| | 供应链后端 | Python（建议） | **统一用Python** |
| **第2层·能力** | 智能体平台 | Python | **统一用Python** |
| | 三方集成 | Python（模型/内部系统）+ Node（电商API可选） | Python为主 |
| **第1层·基建** | 数据库 | PostgreSQL 15 + ClickHouse + Redis 7 | **统一** |
| | 消息队列 | RabbitMQ（一期）→ Kafka（规模扩大） | **统一** |
| | 对象存储 | MinIO + 阿里云OSS | **统一** |
| | 监控 | Prometheus + Grafana + ELK | **统一** |
| | 容器化 | Docker + Docker Compose（一期）→ K8s（规模扩大） | **统一** |

**技术栈统一原则**：
- **后端统一Python**：ML/AI生态最强，团队已有Python能力，跨子系统复用代码库方便
- **前端门户统一Vue3**：门户是唯一入口必须统一；业务应用前端允许React/Vue自选
- **基础设施全统一**：数据库/消息/存储/监控/容器化全体系一致，降低运维成本
- **新子系统建议Vue3**：供应链等新建子系统建议前端也用Vue3，逐步收敛前端技术栈

### 7.5 数据所有权与隔离

| 数据域 | 数据所有者 | 其他子系统访问方式 |
|--------|-----------|-------------------|
| 用户/组织/权限 | 第1层IAM | 只读API |
| 商品SKU主数据 | MDM（源头ERP） | 只读API / 数据同步 |
| 资讯/课程/需求/排名/积分 | AI门户 | 只读API |
| 内容素材/采集数据/爆品预测 | 内容工厂 | API + 事件订阅 |
| 库存/动销/采购/供应商 | 供应链 | API + 事件订阅 |
| 模型调用日志 | 三方集成 | 只读API |
| Agent/工作流/知识库 | 智能体平台 | API调用 |

**原则**：每个数据域有唯一所有者，其他子系统通过API或事件订阅访问，不直接读对方的数据库。
