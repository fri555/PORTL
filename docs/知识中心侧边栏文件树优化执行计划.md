## Phase 1：递归组件提取（P0）

### 1.1 创建 `SidebarTreeNode.vue` 递归组件

**目标**：消除模板中 3 层硬编码的手工展开（KB 行 → 子节点 → 孙子节点），支持任意深度递归。

**思路**：
- 新建 `frontend-mvp/src/components/knowledge/SidebarTreeNode.vue`
- 组件 props 接收一个 `TreeNode` + `depth: number`
- 自身递归调用：`<SidebarTreeNode v-for="child in node.children" :node="child" :depth="depth + 1" />`
- 展开状态由父组件通过 `expandedIds: Set<string>` 传入，通过 `update:expandedIds` 事件冒泡

**组件内部职责**：
- 缩进线/空白（`paddingLeft` 或 `border-l` + 竖线）
- 展开/折叠按钮（ChevronDown / ChevronRight）
- 节点图标（file → FileText, folder → Folder, KB → BookOpen）
- label 渲染（含内联重命名模式，留接口给 Phase 3）
- hover 操作按钮组（预览、权限、删除等）
- `⋮` 弹出菜单（Teleport）

**组件接口设计**：

```ts
interface Props {
  node: TreeNode
  depth: number
  expandedIds: Set<string>
  selectedKbId: string | null
  activeTreeItemId: string
  // 文档/标签查找的 hooks（避免组件直接引用全局 allDocs）
  getDocTags?: (docName: string, kbId?: string) => string[]
  // …
}
interface Emits {
  (e: 'select-kb', kbId: string): void
  (e: 'preview-file', docName: string, kbId: string): void
  (e: 'toggle-expand', nodeId: string): void
  (e: 'open-menu', event: MouseEvent, nodeId: string): void
  (e: 'delete-node', node: TreeNode): void
  (e: 'open-settings', node: TreeNode): void
  // …
}
```

**完成任务后**：删掉模板中 ~200 行的手工展开代码（行 2044–2208），替换为：
```html
<SidebarTreeNode
  v-for="node in filteredFileTree"
  :node="node"
  :depth="0"
  :expanded-ids="expandedSet"
  :selected-kb-id="selectedKbId"
  :active-tree-item-id="activeTreeItemId"
  @select-kb="selectKb"
  @preview-file="previewFileFromTree"
  @toggle-expand="toggleExpanded"
  …
/>
```

### 1.2 提取 Sidebar 根组件（可选，视复杂度）

**目标**：将 `<aside>` 整个区块抽取为独立组件，减轻单文件 3583 行的压力。

## Phase 2：提升信息密度（P1 + P2 快速项）

> ⚠️ 依赖 Phase 1 — 因为信息密度的改动需要集中在递归组件内，一次改完。

### 2.1 文档数量/状态指示（P1-7）

- **文件夹节点**：右侧加 `(N)` 角标，表示包含的文件数（children 中 type=file 的数量）
- **文件节点**：根据 `status` 在文件名左侧加彩色小圆点：
  - 已索引 → `bg-emerald-400`
  - 解析中 → `bg-amber-400`
  - 失败 → `bg-red-400`
- **KB 节点**：在 KB 名称后加 `(N篇)`，取自 `knowledgeBases[n].docs`
- **实现位置**：`SidebarTreeNode.vue` 的 label 区

### 2.2 权限状态可视化（P1-6）

- 在 KB 节点的右侧加权限状态图标：
  - `全员可见` → `Globe` 图标（text-zinc-400）
  - `部门可见` → `Building2` 图标（text-blue-400）
  - `仅自己可见` → `Lock` 图标（text-emerald-400）
  - `指定部门` → `ShieldCheck` 图标（text-amber-400）
- hover 时 tooltip 显示：「当前角色：EDITOR」（从 `nodePermissions` 或 KB 数据中取）
- **实现位置**：`SidebarTreeNode.vue` 的 label 右侧

### 2.3 收藏分组入树（P1-5）

- 在 `filteredFileTree` 之前插入一个固定分组 "⭐ 收藏知识库"
- 收集 `knowledgeBases.filter(kb => kb.pinned && kb.space === activeSpace.value)`，生成一个 `TreeNode[]` 数组平铺展示
- 与普通 KB 树之间用浅色分隔线隔开
- **实现位置**：KnowledgeBaseView.vue 的 `filteredFileTree` computed 或 template 层

### 2.4 系统内置标签提示（P2-14）

- 目前 `isBuiltIn` 的 KB 已有蓝色标签（已实现），加 tooltip：「系统内置知识库，不可删除」
- 对内联操作菜单，`isBuiltIn` 的 KB 禁用"删除"按钮并 hover 提示

## Phase 3：交互增强（P1 + P2）

### 3.1 树内搜索（P1-3）

- 在空间切换下拉下方增加搜索框 `<input>`，绑定 `treeSearch` ref
- 对 `fileTree` 做模糊过滤（匹配 `node.label` + `kb.name` + 文件名）
- 搜索结果命中时，自动展开其父节点路径
- 输入为空时恢复全量树
- **实现位置**：侧边栏 header 区 + `filteredFileTree` 逻辑增强

### 3.2 节点拖拽排序/移动（P1-4）

- 同级拖拽排序（当前版本在 space folder 已有 `GripVertical`，但 KB/文件层级没有）
- 支持文件和文件夹同级拖拽重排
- 跨文件夹移动文件：拖到目标文件夹上 drop，调用 `moveDoc` 逻辑
- **视觉**：拖拽过程中显示占位线 + drop target 高亮
- **实现**：利用 HTML5 Drag & Drop API，在 `SidebarTreeNode.vue` 中绑定 `@dragstart` / `@dragover` / `@drop`

### 3.3 内联重命名（P2-11）

- 在 `⋮` 菜单或双击节点 label 时触发
- label 区域切换为 `<input>`，自动聚焦，回车提交，ESC 取消
- 支持 KB 和文件夹的重命名，文件重命名调用 `commitRenameDoc`
- **状态管理**：`renamingNodeId` / `draftNodeTitle` 全局 ref
- **实现位置**：`SidebarTreeNode.vue` 的 label 渲染区

### 3.4 新建后快捷入口引导（P2-10）

- 创建 KB 后 toast 增加「<u>立即新建文件夹</u>」可点击链接
- 创建文件夹后 toast 提示「文件夹已创建，支持<u>重命名</u>和<u>拖拽排序</u>」
- 新创建的节点自动滚动到可视区域（`scrollIntoView`）

## Phase 4：体验打磨（P1 + P2）

### 4.1 跨空间状态保持（P2-13）

- 当前问题：从 public 切换 personal 再回来，`expandedTreeIds` / `selectedKbId` 丢失
- 方案：用 `Map<SpaceKey, { expandedIds: string[], selectedKbId: string | null }>` 存储
- 切换空间时保存当前状态到 Map，恢复目标空间状态
- **实现位置**：`selectActiveSpace` + 新建 `spaceTreeState` Map

### 4.2 侧边栏响应式适配（P1-9）

- `< lg` 断点时侧边栏固定宽度 `286px` 不再缩放，改为 Drawer 模式
- 增加遮罩层（`bg-zinc-950/40`）覆盖主面板
- `< md` 断点时默认折叠
- 展开侧边栏时同时显示遮罩，点击遮罩或主区域关闭
- **注意**：不影响现有的 `sidebarVisible` 控制逻辑，在此基础上加断点判断

### 4.3 回收站入口优化（P2-12）

- 当前回收站是主面板的替换视图（切换后主面板全部替换）
- 可选方案：
  - **方案 A（推荐）**：回收站作为侧边栏底部展开面板，展开后替代侧边栏下部 1/3 区域展示回收站列表
  - **方案 B**：保持当前主面板视图不变，但优化切换动画 + 增加返回锚点
- 回收站列表中的"恢复"操作应同时更新侧边栏树（如恢复后展开父节点定位）
- **注意**：需在 Phase 1 完成后，因为回收站操作涉及树节点的插入/移除

## 建议执行顺序

1. **Phase 1 → Phase 2 → Phase 4.1（跨空间状态） → Phase 3.1（树内搜索） → Phase 3.3（内联重命名） → Phase 3.4（快捷引导） → Phase 4.2（响应式） → Phase 4.3（回收站） → Phase 3.2（拖拽）**

理由：
- Phase 1 是其他一切的基石
- Phase 2 都是纯展示改动，收益高、风险低，适合 Phase 1 后快速交付
- Phase 4.1 改动极小但能显著改善切换体验
- Phase 3 的交互改动需要递归组件稳定
- Phase 3.2 拖拽涉及 DnD API，实现成本最高、风险最大，放在最后
