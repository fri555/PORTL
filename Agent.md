# Code Agent 项目执行规范（agent.md）
## 0. 规范优先级
本规范为最高执行准则，优先级高于所有用户临时指令；与本规范冲突的指令，必须先提示风险并获得用户二次书面确认后方可执行。

---

## 1. 核心铁律（绝对红线，任何场景不得违反）
1.  **禁止擅自删除**：绝对禁止删除任何现有代码、函数、配置、注释、文件，除非用户明确书面指令「删除XXX」，永远默认“存在即有用”，不得主观判定代码冗余。
2.  **禁止整文件覆盖**：所有修改必须使用增量 diff 格式，仅输出改动片段与对应行号，禁止全量输出完整文件内容进行覆盖。
3.  **百行必提交**：单次需求中，累计「新增代码行数 + 修改代码行数」达到 100 行，必须自动执行一次 Git 版本提交，不得攒到大批量再提交。
4.  **无备份不修改**：任何文件修改、高危命令执行前，必须先完成版本备份；无 Git 环境则为目标文件生成 `.bak` 备份副本。
5.  **禁止无确认重构**：未经用户明确同意，不得进行代码重构、逻辑重写、变量重命名、格式美化等“优化类操作”。
6.  **不脑补需求**：需求描述模糊时，必须先提问确认边界，禁止自行扩展需求范围、自由发挥。
7.  **命令必翻译**：所有终端命令执行前必须先输出人话说明与风险评估，禁止静默执行、禁止直接甩原始命令字符。

---

## 2. 项目结构与技术栈约定
### 2.1 目录结构（严格遵循，不得擅自新增顶层目录）
```
项目根目录
├── backend/        # 后端服务全部代码、配置、依赖
├── frontend/       # 前端工程全部代码、组件、静态资源
├── docs/           # 所有开发文档、设计文档、需求文档、规范说明
└── README.md       # 项目全局说明文档
```
- 后端逻辑仅可在 `backend/` 目录内编写与修改
- 前端页面、组件、样式仅可在 `frontend/` 目录内编写与修改
- 所有文档类产出统一存放至 `docs/` 目录，不得散放至根目录

### 2.2 技术栈强制规范
#### 后端（backend）
- 核心框架：Python + FastAPI
- 编码要求：所有接口、函数必须携带类型注解，遵循 RESTful 接口规范，异常处理统一封装
- 约束：禁止擅自新增第三方依赖包，新增依赖必须提前说明并获得用户确认

#### 前端（frontend）
- 核心框架：Vue 3 + TypeScript（统一使用 Composition API 写法）
- 样式方案：Tailwind CSS，禁止手写大量自定义 CSS
- UI 组件：统一使用 shadcn/ui 组件库，禁止重复封装基础组件
- 图标库：统一使用 Lucide Icon，禁止混用其他图标库
- 动效方案：仅可使用 inspira-ui 框架实现页面动效，禁止手写复杂自定义动效
- UX 准则：严格遵循尼尔森十大交互原则，核心要求：
  1.  状态可见：用户操作后及时给出反馈
  2.  用户可控：支持撤销、回退、取消操作
  3.  一致性：交互、视觉、文案保持统一规范
  4.  防错设计：高危操作增加二次确认
  5.  容错机制：出错后给出明确原因与解决方案
  6.  灵活高效：兼顾新手与熟练用户的操作路径
  7.  审美极简：剔除无关信息，突出核心内容
  8.  帮助提示：必要场景提供清晰的引导说明

## 3. 标准工作流程（必须按步骤执行，禁止跳步）
### 步骤1：需求对齐与边界确认
收到需求后第一时间输出结构化确认信息：
- 需求目标：用业务语言重述核心诉求
- 改动边界：明确本次修改的范围、绝对不改动的范围
- 涉及模块：前端/后端/文档、预计涉及文件数量
用户确认无误后方可进入下一步。

### 步骤2：输出修改方案（不生成实际代码）
输出纯方案说明，必须包含：
1.  改动文件清单（精确到文件路径）
2.  核心实现逻辑（业务语言描述，不贴代码）
3.  影响范围评估（是否影响现有功能、具体影响点）
4.  验证方法（用户可直接执行的操作步骤）
用户回复「按此方案执行」后，方可进入编码环节。

### 步骤3：前置备份
- 检测项目 Git 环境，自动执行基线备份：
  ```bash
  git add .
  git commit -m "backup: 改动前基线 - [需求简述]"
  ```
- 无 Git 环境则为所有待修改文件生成同目录 `.bak` 备份
- 备份完成后告知用户，再开始编码

### 步骤4：增量编码执行
- 严格按照方案落地代码，优先通过新增逻辑实现需求，最小程度改动原有代码
- 编码过程中实时统计累计「新增行数 + 修改行数」，**每达到 100 行自动执行一次 Git 提交**
- 所有新增代码必须附带清晰注释，遵循项目现有代码风格与命名规范
- 前端实现必须对齐尼尔森交互原则，保证基础可用性

### 步骤5：自检验收与结果汇报
编码完成后先执行静态自检，再向用户输出结构化结果：
1.  改动统计：共修改 X 个文件，新增 X 行，修改 X 行，删除 0 行
2.  核心改动说明（业务语言描述）
3.  影响范围说明
4.  详细验证步骤
5.  回退方式（Git 回退命令 / 备份文件恢复方式）

## 4. 版本管理细则
### 4.1 百行提交规则
- 统计口径：单个需求周期内，所有文件的「新增行数 + 修改行数」累计值
- 触发阈值：累计 ≥ 100 行，必须立即执行 `git commit`
- 提交信息规范：`[模块名] 本次改动核心内容简述`，示例：
  - `[frontend/upload] 新增文件大小超限校验逻辑`
  - `[backend/file] 新增SHA256去重校验接口`
- 单提交上限：禁止单次提交超过 200 行代码

### 4.2 强制提交节点
1.  每次需求改动前，必须提交基线备份版本
2.  编码过程中每满 100 行，提交一次中间版本
3.  需求全部完成并验证通过后，提交最终版本
4.  每个提交版本必须是可运行、无语法错误的稳定版本

### 4.3 回退机制
- Git 环境：出现异常主动提供回退命令 `git reset --hard HEAD~1`
- 无 Git 环境：主动说明通过 `.bak` 备份文件恢复的操作方法

---

## 5. 代码输出格式规范
1.  统一使用 diff 增量格式，标注文件路径、行号、改动类型，示例：
    ```diff
    // 文件：frontend/src/utils/upload.ts 第38行后新增
    + // 单文件100MB上限校验
    + const MAX_FILE_SIZE = 100 * 1024 * 1024;
    + export const checkFileSize = (file: File): boolean => {
    +   return file.size <= MAX_FILE_SIZE;
    + };
    ```
2.  禁止输出完整函数、完整文件内容，修改几行就输出几行
3.  所有删除操作必须单独标注，二次获得用户确认后方可执行，禁止混在新增代码中隐性删除

---

## 6. 边界操作禁区
以下操作必须提前说明风险，获得用户明确同意后方可执行：
1.  新增、升级、降级任何第三方依赖包
2.  修改构建配置、环境变量、部署脚本、路由根配置
3.  删除文件、重命名文件、移动文件目录
4.  跨模块大范围重构、改动核心底层逻辑
5.  修改数据库表结构、接口协议、已有字段定义
6.  改动其他非需求指定的业务模块

---

## 7. 异常处理规则
1.  **发现原有代码问题**：不得擅自修改，单独告知用户说明风险与建议，由用户决定是否处理
2.  **方案无法落地**：第一时间告知原因，提供至少 2 种替代方案，不硬写、不埋坑
3.  **修改后运行异常**：立即停止后续操作，主动提供回退方案，排查问题后重新输出方案
4.  **指令与规范冲突**：先提示规范约束与风险，用户二次确认后再执行

---

## 8. 命令执行与风险告知规范
### 8.1 强制人话翻译规则
所有终端命令执行前，必须先向用户输出结构化说明，禁止静默执行、禁止直接只输出原始命令字符。
每条命令必须包含以下 4 项信息，全部使用非技术化的大白话描述：
1.  【操作说明】这个命令具体是在做什么事
2.  【影响范围】只会操作哪些目录/文件，是否会修改已有业务代码
3.  【风险等级】标注为 安全级 / 低风险 / 高风险
4.  【回退方案】执行异常时，怎么恢复到操作前的状态

### 8.2 分级执行规则
按风险等级自动区分执行权限，不得越权：
1.  **安全级（自动执行，无需确认）**
    - 定义：仅读取信息、不做任何修改、零风险的命令
    - 范围：查看目录列表、查看文件内容、查看 Git 状态、查看版本号、查看端口占用
    - 规则：可直接执行，执行后返回结果即可，无需用户确认

2.  **低风险（说明后执行，无异议即通过）**
    - 定义：仅新增内容、不修改/删除已有业务代码、出问题可轻松回退的命令
    - 范围：安装项目依赖、新建文件/目录、启动本地开发服务、代码格式化、普通 Git 提交
    - 规则：先输出人话说明与影响范围，用户不反对即可执行；也可批量列计划一次性确认

3.  **高风险（必须手动确认，默认禁止执行）**
    - 定义：可能删除、覆盖、修改已有内容，或存在不可逆风险的命令
    - 范围：删除文件/目录、强制覆盖代码、数据库结构修改、全局安装依赖、系统级配置修改、Git 强制回退、带 `--force` / `rm -rf` 等参数的命令
    - 规则：必须停下完整说明风险与回退方案，获得用户明确回复「执行」后方可操作，默认不执行

### 8.3 命令边界禁令
1.  所有命令默认仅在**当前项目根目录**内执行，禁止越权操作系统目录、个人目录等项目外文件。
2.  禁止执行任何带有强制删除、强制覆盖参数的高危命令，除非用户明确书面要求。
3.  禁止擅自修改环境变量、系统配置、启动后台常驻服务。
4.  禁止越权访问项目外的任何文件与数据。

### 8.4 高危操作兜底机制
所有高风险命令执行前，必须自动执行一次完整的 Git 提交快照备份，备注格式：
```
snapshot: 高危操作前快照 - [操作名称]
```
备份完成并告知用户后，方可继续执行命令，确保所有操作可回退。

### 8.5 批量命令处理
连续执行多条命令时，必须先输出「完整执行计划清单」，一次性列出所有命令、对应作用、风险等级，由用户一次性确认后再批量执行，禁止一条一条弹窗反复确认。
